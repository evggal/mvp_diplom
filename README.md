## Требования к продукту

Необходимо реализовать систему для дискретно-событийного моделирования, где система представляется в виде графа, в котором вершины - точки массивого обслуживания, рёбра - возможные пути следования заявок. 
Особенностью является то, что состояние системы изменяется по средствам наступления событий в системе. 
События: 
 - появление заявки
 - заявка пришла на точку обслуживания
 - заявка начала обслуживание
 - заявка закончила обслуживание
 - заявка вышла из системы

В системе время отсчитывается по событиям, при помощи колендаря событий, в котором зрянятся все предыдущие и последующие известные на данный момент события.

Заявки генерируются на входе (генераторе заявок) и заканчивают свой путь на выходе из системы
Заявки следуют определённому пути, который определяется соответсвующими связями между вершинами, есть входящие и выходящие связи сооьветсчтвенно.
Для каждой точки массового обслуживания необходимо предусмотреть очередь с приоритетом по времени прихода заявки на точку обслуживания.

Необходимо реализовать настраевоемый генератор заявок с различными распределениями.
Время обрботки заявки на точке обслуживания распределяется по нормальному закону, но можно предусмотреть выбор и других распределений.

Всё, что связано с законами распределения необходимо вынести в отдельный модуль.

Так же при моделировании необходимо собрать все возможные метрики для анализа:
 - Размер очереди для каждой точки
 - Время ожидания в очереди
 - Загруженность точек
 - И другие

На фронтенде необходимо реализовать редактов модели, через которую пользователь сможет создавать модель, назначать связи, устанавливать законы распределения и тд. После чего по кнопоске "Моделирование" модель будет отправляться на сервер, моделироваться, после моделирования должна появиться кнопочка "Результаты моделирования", по которой можно будет посмотреть (скорее всего на другой странице) пошуговое моделирование, графики и различные метрики.
На время проведения моделирования сам сайт (фронтенд) не должен зависать, пользователь должен иметь возможность продолжать работу, пока сервер производит моделирование.

Результаты моделирования необходимо сохранять в отдельную папку для каждого моделирования, у пользователя необходимо запрашивать название модели, внутри папки моделирования должен быть csv файл по которому можно будет провести визуализацию процесса моделирования, csv файл с метриками, по которым можно будет воспроизвести графики и json со структурой модели, по которой можно будет отрисовать модель на фронтенде. Так же необходимо выводить список моделей пользователю, чтобы он мог переключаться от одной к другой, просматривая про каждую все необходимые данные.

## Требования к структурам

Точка массового обслуживания:
 - Время открытия и закрытия точки (расписание задаётся перед началом моделирования)
 - Выбор закона распределения и его параметров
 - Выбор количество каналов обслуживания
 - Список/выбор дельнейших путей следования заявок с некоторыми вероятностями
 - Реализация очереди для точки обслудивания

Ребро:
 - Перемещение по рёбрам должно занимать определённое количество времени, так же распределённое по нормальному закону (или другим законам)

## Требования к функционалу

Запус моделирования происходит через фронтенд, почле чего сервер запускает моделирование, результат моделирования записывается в csv файл, после чего данные из файла отправляются на фронтенд для отрисовки. 

В папке Modeling должно находится всё, что связано с моделированием. Все структуры должны быть вынесены в отдельный модуль. 

В папке api  должно лежать fastapi приложение.

Для управления python и библиотеками используй uv

В папке frontend должно лежать ReactTS приложение. С граммотной архитектурой, разделением страниц, элементов, интерфейсов


## Требования к коду

Необходи писать простой и понятный, читаемый код, чтобы в нём было просто разобраться, необходимо в меру комментировать написанный код. Классы и функции не должны быть слишком большими, не должно быть слишком длинных строк без резкой необходимости. Код должен быть чистым и понятным, с понятными и адекватными названиями переменных. Используй правила: все функции называются с большой букывы, все переменные с маленькой, если переменная должна состоять из нескольких слов разделяй их _ (нижним подчёркиванием), в React для названий Hook применяй стиль где первый символ маленький, а дальше верблюжий шрифт (каждое новое слове без разделителя, но с большой буквы)
