# Глава 2. Методы реализации программного продукта

Во второй главе рассматриваются методические решения, принятые при проектировании программного продукта для дискретно-событийного моделирования системы обслуживания аэропорта.  
Цель главы — показать, какими способами, на каких технологических основаниях и на базе каких данных формируется модель, а также как выполняется интерпретация полученных результатов.

Структура главы соответствует развёрнутому плану ВКР:

1. Инструменты для реализации программного продукта.
2. Данные для моделирования системы аэропорта.
3. Методы анализа полученных результатов.

---

## 2.1 Методологические основы реализации программного продукта

### 2.1.1 Системный подход к построению решения

Разработка выполнена на основе системного подхода: модель аэропорта рассматривается как совокупность взаимосвязанных подсистем обслуживания, а программный продукт — как единая информационно-вычислительная система, состоящая из независимых, но согласованных модулей.

В рамках подхода выделены три уровня:

1. Концептуальный уровень — формализация предметной области в виде сети массового обслуживания.
2. Алгоритмический уровень — описание состояния системы через события и календарь событий.
3. Программный уровень — реализация вычислительного ядра, API и пользовательского интерфейса.

Такое разделение уменьшает сложность разработки и позволяет отдельно совершенствовать:

- математическую модель;
- алгоритмы имитации;
- интерфейс и визуализацию;
- способы хранения и анализа результатов.

### 2.1.2 Графовое представление аэропорта как СМО

Для формализации пассажиропотока используется ориентированный граф:

- вершины графа соответствуют узлам обслуживания;
- рёбра задают допустимые переходы между узлами;
- специальные типы узлов отражают источник потока и выход из системы.

Поддерживаются типы узлов:

- `generator` — генератор входящих заявок;
- `service` — обслуживающий узел;
- `exit` — выход из системы.

Каждый обслуживающий узел имеет набор параметров:

- время открытия и закрытия;
- количество каналов обслуживания;
- закон распределения времени обслуживания;
- набор исходящих маршрутов с вероятностями перехода.

Маршрутизация после обслуживания задаётся вероятностно, а сумма вероятностей исходящих рёбер для каждого невыходного узла нормируется к 100%.  
Это позволяет задавать как линейные, так и разветвлённые сценарии прохождения пассажиров.

### 2.1.3 Дискретно-событийный метод имитации

Ключевой метод расчёта — дискретно-событийное моделирование.  
Время в системе изменяется не непрерывно, а скачкообразно — только в моменты наступления событий.

Событийная модель включает следующие классы событий:

1. Генерация заявки.
2. Прибытие заявки в узел.
3. Начало обслуживания.
4. Завершение обслуживания.
5. Выход заявки из системы.
6. Внутренние события перепроверки доступности узла (по расписанию работы).

Преимущества метода для аэропортовой задачи:

- высокая вычислительная эффективность при большом горизонте времени;
- естественное описание очередей и многоканального обслуживания;
- удобное расширение за счёт новых типов узлов и законов распределения;
- получение детализированного журнала событий для последующего анализа.

### 2.1.4 Модульный метод проектирования

Архитектура продукта построена по модульному принципу с разделением ответственности:

- модуль схем данных и валидации конфигурации;
- модуль генерации случайных величин;
- модуль имитационного ядра;
- модуль API;
- модуль хранения и загрузки результатов;
- модуль визуального редактора и аналитики.

Преимущества модульного подхода:

1. Локализация изменений: доработка алгоритмов не требует переписывания интерфейса.
2. Переиспользование: один и тот же модуль распределений работает для генератора, узлов и рёбер.
3. Тестируемость: отдельные компоненты можно проверять изолированно.
4. Масштабируемость: возможно добавление новых метрик и новых типов узлов.

### 2.1.5 Метод асинхронного запуска расчёта

Расчёт моделирования отделён от пользовательского интерфейса через асинхронный запуск задач:

- клиент инициирует запуск;
- сервер ставит задачу в очередь;
- статус работы отслеживается опросом;
- по завершении доступны результаты конкретного прогона.

Такой метод предотвращает блокировку фронтенда и позволяет пользователю параллельно работать с системой при длительных расчётах.

## 2.2 Инструменты для реализации программного продукта

### 2.2.1 Критерии выбора инструментов

Инструментарий подбирался по следующим критериям:

1. Поддержка численных расчётов и работы с вероятностными моделями.
2. Скорость разработки и читаемость кода.
3. Простота локального развёртывания.
4. Возможность разделения backend/frontend.
5. Наличие удобных библиотек визуализации и маршрутизации интерфейса.

### 2.2.2 Инструменты backend-части

Серверная часть реализована на Python с использованием следующих компонентов:

- **FastAPI** — построение HTTP API и валидация входных/выходных схем;
- **Pydantic** — строгая типизация и проверка структуры конфигурации модели;
- **NumPy** — генерация случайных величин и статистические расчёты;
- **Pandas** — формирование и сериализация таблиц событий и метрик;
- **Uvicorn** — ASGI-сервер для запуска приложения;
- **PyJWT** — выпуск и проверка токенов авторизации;
- **uv** — управление зависимостями и запуск backend-приложения.

Причины выбора:

- Python удобен для имитационного моделирования и прототипирования вычислительных алгоритмов;
- FastAPI обеспечивает быстрый запуск REST-сервиса и хорошо сочетается с Pydantic;
- NumPy/Pandas дают готовую базу для статистики и хранения табличных результатов.

### 2.2.3 Инструменты frontend-части

Клиентская часть реализована на React + TypeScript:

- **React** — компонентная организация интерфейса;
- **TypeScript** — типобезопасность модели данных и API-контрактов;
- **React Router** — маршрутизация страниц редактора, результатов и списка моделей;
- **Axios** — HTTP-взаимодействие с API;
- **Chart.js** и `react-chartjs-2` — построение графиков метрик;
- **Vite** — быстрая сборка и режим разработки.

Причины выбора:

- React упрощает построение интерактивного визуального редактора графа;
- TypeScript снижает риск ошибок на стыке фронтенда и бэкенда;
- Chart.js закрывает задачу аналитической визуализации без избыточной сложности.

### 2.2.4 Организация хранения результатов

Для каждого прогона формируется отдельный каталог с артефактами:

- `model.json` — конфигурация модели;
- `events.csv` — журнал событий;
- `metrics.csv` — агрегированные метрики по узлам;
- `summary.json` — сводные показатели прогона.

Метод хранения «один прогон — одна папка» позволяет:

1. Повторно воспроизводить расчёт.
2. Сравнивать сценарии между собой.
3. Восстанавливать модель для режима «взять за основу».
4. Сохранять трассировку событий для отладки и анализа.

### 2.2.5 Средства обеспечения качества и надёжности

На уровне инструментов надёжность обеспечивается следующими практиками:

- строгая валидация конфигурации перед запуском;
- единые типы данных на клиенте и сервере;
- защита API через JWT-авторизацию;
- контроль целостности результатов при сохранении;
- модульная структура кода с читаемыми именами и изолированными функциями.

## 2.3 Данные для моделирования системы аэропорта

### 2.3.1 Категории входных данных

Для построения модели используются четыре группы данных.

**1. Общие параметры моделирования**

- длительность моделирования;
- ограничение по количеству заявок;
- начальное значение генератора случайных чисел (seed).

**2. Параметры генератора входного потока**

- узел входа в систему;
- время старта и остановки генерации;
- закон генерации заявок и его параметры.

**3. Параметры узлов обслуживания**

- идентификатор и название узла;
- тип узла;
- время открытия/закрытия;
- число каналов обслуживания;
- закон распределения времени обслуживания.

**4. Параметры маршрутов и рёбер**

- наименование ребра;
- целевой узел перехода;
- вероятность перехода;
- распределение времени перемещения между узлами.

### 2.3.2 Структурирование данных в конфигурацию модели

Все входные данные формируются в единую конфигурацию, включающую:

- `SimulationConfig`;
- `GeneratorConfig`;
- массив `NodeConfig`;
- массив `EdgeConfig`.

Маршрутная часть задаётся внутри узлов через массив `routes`, что позволяет хранить логику переходов рядом с параметрами узла-источника.

Метод такого структурирования обеспечивает:

1. Логическую целостность модели.
2. Простую сериализацию в JSON при передаче на сервер.
3. Однозначную валидацию до начала расчёта.

### 2.3.3 Распределения, применяемые в данных модели

В рамках модели поддерживаются следующие законы.

**Для генератора заявок:**

- пуассоновский;
- экспоненциальный;
- детерминированный;
- Эрланга;
- интервальный;
- по интенсивности.

**Для обслуживания в узлах:**

- нормальный;
- равномерный;
- экспоненциальный;
- гиперэкспоненциальный;
- детерминированный;
- Эрланга.

**Для рёбер (перемещение):**

- нормальный;
- равномерный;
- экспоненциальный;
- детерминированный;
- Эрланга.

Выбор распределений основан на необходимости покрыть:

- стационарные потоки;
- потоки с фиксированным интервалом;
- процессы с выраженной вариативностью времени обслуживания;
- комбинированные случаи с «быстрым» и «медленным» режимами обработки.

### 2.3.4 Источники и подготовка исходных параметров

Параметры модели могут формироваться из:

1. Наблюдений и фактической статистики работы аэропорта.
2. Нормативов и регламентов по времени операций.
3. Экспертных оценок специалистов предметной области.
4. Сценарных предположений для «что-если» анализа.

Рекомендуемый порядок подготовки данных:

1. Определить состав узлов и маршрутов пассажира.
2. Оценить интенсивность входа по временным интервалам.
3. Настроить законы обслуживания по типам операций.
4. Задать вероятности развилок на маршрутах.
5. Проверить единицы измерения (в проекте — минуты).
6. Провести валидацию суммы вероятностей и расписаний работы узлов.

### 2.3.5 Ограничения и допущения данных

Для практического использования модели важно фиксировать принятые допущения:

- структура графа считается заданной на время одного прогона;
- правила маршрутизации не меняются в процессе прогона;
- параметры распределений стационарны в пределах сценария;
- поведение отдельных заявок считается независимым, если иное не задано структурой очередей;
- качество результатов зависит от полноты и корректности исходных параметров.

Явная фиксация допущений повышает интерпретируемость итогов и корректность управленческих выводов.

## 2.4 Методы анализа полученных результатов

### 2.4.1 Метод метрик производительности СМО

Для оценки работы модели используются метрики двух уровней.

**Уровень системы:**

- количество созданных заявок;
- количество завершённых заявок;
- число заявок, оставшихся в системе к концу горизонта;
- среднее время пребывания в системе;
- пропускная способность;
- количество обработанных событий.

**Уровень узлов:**

- число поступивших/начатых/завершённых заявок;
- средняя и максимальная длина очереди;
- среднее время ожидания;
- среднее время обслуживания;
- коэффициент загрузки канала(ов).

Методика позволяет одновременно анализировать:

- глобальную пропускную способность;
- локальные узкие места;
- устойчивость работы отдельных этапов обслуживания.

### 2.4.2 Метод поэтапной интерпретации событий

Помимо агрегированных метрик применяется анализ трассы событий:

1. Отслеживается путь каждой заявки по узлам.
2. Фиксируются моменты возникновения и спада очередей.
3. Анализируются интервалы перегрузки по времени.
4. Сопоставляются всплески нагрузки с параметрами генерации и расписания.

Преимущество метода — возможность объяснить, **почему** возникли конкретные значения метрик, а не только констатировать их итоговые величины.

### 2.4.3 Метод визуального аналитического контроля

Для прикладной интерпретации используются:

- анимированная схема движения заявок;
- графики по всем узлам системы;
- графики по выбранному узлу во времени;
- таблица агрегированных метрик.

Визуальный метод позволяет:

1. Быстро обнаруживать «узкие» вершины.
2. Сравнивать состояние очередей между этапами обслуживания.
3. Проверять корректность маршрутов и распределений.
4. Проводить экспертную валидацию поведения модели.

### 2.4.4 Сценарный метод сравнения вариантов

Оптимизационный анализ выполняется итеративно:

1. Фиксируется базовый сценарий.
2. Меняется один класс параметров (например, число каналов на выбранном узле).
3. Выполняется новый прогон.
4. Сравниваются изменения ключевых KPI.
5. Делается вывод об эффекте модификации.

Основной принцип — изменять параметры дозированно и сравнивать результаты на сопоставимых условиях (в том числе с фиксированным `seed` при необходимости воспроизводимости).

### 2.4.5 Метод выявления узких мест

Узел рассматривается как потенциально проблемный при сочетании признаков:

- высокая загрузка каналов;
- растущая средняя очередь;
- высокий максимум очереди;
- рост среднего времени ожидания;
- эффект «переноса перегрузки» на последующие узлы.

После выявления узких мест формируется набор управленческих воздействий:

- увеличение числа каналов;
- изменение расписания работы узла;
- перераспределение маршрутных вероятностей;
- корректировка интенсивности входного потока;
- изменение параметров распределения обслуживания.
